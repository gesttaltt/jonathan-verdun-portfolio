#### H1 — `layout.tsx` at 0 % coverage

**File:** `src/app/layout.tsx`  
**Severity:** High  
**Category:** Test Coverage

`layout.tsx` is the most consequential file in the application — it provides the skip-nav
link, the JSON-LD Person schema, the font class, the HTML `lang` attribute, and both
context providers (`MotionProvider`, `ProjectProvider`). Despite this, it has 0 % coverage
across all metrics.

The file is excluded from the coverage report by default because Next.js server components
render in Node, not jsdom, but it is not listed in `coveragePathIgnorePatterns` — meaning
the zeros appear in the table and drag down the aggregate. More importantly, the semantic
elements and data it injects are untested.

**Testable assertions that are currently missing:**

- `<html lang="en">` attribute
- Skip-nav link renders and is `sr-only` until focused
- `<script type="application/ld+json">` is present in the document
- JSON-LD content includes `name`, `jobTitle`, `url`, `email`, `sameAs`
- `font-mono antialiased` class on `<body>`

**Recommended fix:** Add `src/__tests__/layout.test.tsx` that shallow-renders `RootLayout`
with children and asserts the above. For JSON-LD, assert `document.querySelector(...)`.
Alternatively, add `layout.tsx` to `coveragePathIgnorePatterns` if testing it in jsdom
is architecturally impractical — but the explicit decision should be recorded.

#### H2 — `InteractiveTopology.tsx` and `TopologyMesh.tsx` at 0 % coverage

**Files:** `src/components/InteractiveTopology.tsx`, `src/components/TopologyMesh.tsx`  
**Severity:** High  
**Category:** Test Coverage / Risk Surface

The Three.js canvas (and its shader mesh) are the most visually prominent element of the
portfolio — the animated icosahedron fills the entire background. Yet both files have 0 %
coverage and are not listed in `coveragePathIgnorePatterns`. This means:

1. Refactoring either file can silently break the visual with no test signal.
2. The aggregate coverage numbers are pulled down by two files that legitimately cannot be
   unit-tested in jsdom.

`TopologyMesh.tsx` uses `useFrame`, `THREE.IcosahedronGeometry`, and custom GLSL uniforms —
none of which are mockable in jest/jsdom without substantial infrastructure.

**Recommended fix (two-track):**

_Track A — cover what is testable:_ Add a test for `InteractiveTopology.tsx` that mocks
`@react-three/fiber`'s `Canvas` and asserts the component renders the `aria-hidden="true"`
canvas wrapper and the grid overlay div. This is achievable without a real WebGL context.

_Track B — exclude the untestable:_ Add `TopologyMesh.tsx` (and optionally
`InteractiveTopology.tsx` if mocking is impractical) to `coveragePathIgnorePatterns` in
`jest.config.mjs`. The `TopologyShaders.ts` file (already excluded) sets the precedent.
Explicit exclusion is more honest than implicit 0 % showing in coverage tables.

#### H3 — OG image renders with system `monospace` font, not JetBrains Mono

**File:** `src/app/opengraph-image.tsx`  
**Severity:** High  
**Category:** Visual Consistency / SEO

`opengraph-image.tsx` passes `fontFamily: 'monospace'` as an inline style to `ImageResponse`.
`next/og` renders to a PNG via Satori — it does not inherit browser fonts. The `monospace`
keyword resolves to Satori's internal fallback (usually Noto Sans Mono), which looks visibly
different from JetBrains Mono used throughout the live site. Any social share card will not
match the portfolio's visual identity.

To use JetBrains Mono in the OG image, the font must be fetched as an `ArrayBuffer` and
passed in the `fonts` option of `ImageResponse`:

```ts
const font = await fetch(new URL('/fonts/JetBrainsMono-Bold.woff', request.url))
const fontData = await font.arrayBuffer()

return new ImageResponse(<div ...>, {
  ...size,
  fonts: [{ name: 'JetBrains Mono', data: fontData, weight: 700 }],
})
```

This requires either bundling the font file in `/public/fonts/` or fetching from Google
Fonts at build time. The component would also need to become `async` and accept `request`
as a parameter (standard Next.js OG handler signature).

**Recommended fix:** Add `JetBrainsMono-Bold.woff` to `public/fonts/`, make the handler
`async`, fetch and pass the font buffer to `ImageResponse`.

#### M1 — `QAPhilosophyGrid` uses hardcoded animation values instead of centralized variants

**File:** `src/components/QAPhilosophyGrid.tsx:12-18`  
**Severity:** Medium  
**Category:** Consistency / Maintainability

The previous audit (M1) centralized `FadeInSection` to use `fadeInVariants` from
`animations.ts`. However, `QAPhilosophyGrid` still hardcodes its motion values inline:

```tsx
initial={{ opacity: 0, y: 12 }}
whileInView={{ opacity: 1, y: 0 }}
viewport={{ once: true, margin: '-40px' }}
transition={{ duration: 0.35, ease: 'easeOut', delay: i * 0.12 }}
```

`SectionHeader` similarly hardcodes its own entry animation:

```tsx
initial={{ opacity: 0, y: 10 }}
whileInView={{ opacity: 1, y: 0 }}
transition={{ duration: 0.35, ease: 'easeOut' }}
```

Neither imports from `animations.ts`. There are now three different `y` offsets (10, 12, 20)
and two duration values (0.35, 0.45) spread across the codebase with no shared constant.
If the design direction changes (e.g., faster entrance animations), updates must be made in
multiple places.

**Recommended fix:** Extend `animations.ts` with a `staggerItemVariants(delay: number)`
variant for list-item entrances (y: 12, duration: 0.35) — distinct from `fadeInVariants`
which serves section-level fades. Update `QAPhilosophyGrid` to import and use it.

#### M3 — `ErrorBoundary` has no reset mechanism; requires full page reload to recover

**File:** `src/components/ErrorBoundary.tsx:34-50`  
**Severity:** Medium  
**Category:** UX / Resilience

When the Three.js topology fails (GPU unavailable, WebGL not supported, driver crash), the
`ErrorBoundary` renders a static `"A rendering error occurred. Try refreshing the page."`
message. There is no `resetErrorBoundary` button that calls `this.setState({ hasError: false })`.

The Next.js `error.tsx` page provides a `reset` callback — `ErrorBoundary` should offer
the same pattern for in-page boundaries, especially for optional decorative components like
`TopologyLoader` where a retry is low-cost.

**Recommended fix:** Add a `reset()` method and a retry button to the default fallback UI:

```tsx
reset() {
  this.setState({ hasError: false, error: null })
}

// In the fallback render:
<button onClick={() => this.reset()} className="mt-4 text-xs text-blue-400 hover:text-blue-300">
  Retry
</button>
```

#### M4 — JSON-LD Person schema missing structured fields for richer SEO signals

**File:** `src/app/layout.tsx:40-48`  
**Severity:** Medium  
**Category:** SEO

The current JSON-LD schema includes: `name`, `jobTitle`, `url`, `email`, `sameAs`. This is
valid but minimal. Schema.org's `Person` type supports additional fields that improve
eligibility for Google Knowledge Panel rich results and improve the semantic signal for
academic/research-oriented searches:

| Field        | Value                                                                 | Benefit                 |
| ------------ | --------------------------------------------------------------------- | ----------------------- |
| `knowsAbout` | `['QA Automation', 'Bioinformatics', 'Computational Biology', 'TDD']` | Research intent signals |
| `alumniOf`   | Organization(s) if applicable                                         | Credential signals      |
| `worksFor`   | `{ '@type': 'Organization', name: 'Ai-Whisperers', url: '...' }`      | Affiliation             |
| `image`      | OG image URL                                                          | Rich result avatar      |

**Recommended fix:** Extend the `jsonLd` object in `layout.tsx` with at minimum
`knowsAbout` and `worksFor` (sourced from `siteConfig` to keep it DRY).

#### M5 — `ResearchSpec` / `SystemSpec` type coupling requires updating two places to add a new entry

**Files:** `src/lib/contracts/BioinformaticsContract.types.ts`, `src/components/BioinformaticsResearchCard.tsx`  
**Severity:** Medium  
**Category:** Architecture / Data Coupling

`ResearchSpec.focus` is typed as `'HIV' | 'Codon Encoding'`. `BioinformaticsResearchCard`
maintains a parallel `FOCUS_LABELS` and `FOCUS_DESCRIPTIONS` map keyed on the same union.
Adding a third research area (e.g., `'CRISPR'`) requires three coordinated changes:

1. Extend the union in `BioinformaticsContract.types.ts`
2. Add the new entry in `BioinformaticsContract.ts`
3. Add entries to both `FOCUS_LABELS` and `FOCUS_DESCRIPTIONS` in `BioinformaticsResearchCard.tsx`

TypeScript will catch (3) if the record types are kept exhaustive, but the architecture
still violates the principle that data and its presentation metadata should travel together.

**Recommended fix:** Enrich `ResearchSpec` with optional `label` and `description` fields
populated in the contract data. The card component then just reads `spec.label ?? spec.focus`
and `spec.description`. This removes the parallel map entirely and makes the card generic.

Same pattern applies to `SystemSpec` in `DataEngineeringContract.types.ts` / `SystemSpecCard.tsx`.

#### M6 — `docs/api/` TypeDoc output is committed to the repo

**File:** `.gitignore` (missing entry)  
**Severity:** Medium  
**Category:** Repository Hygiene

`npm run docs` generates ~55 files into `docs/api/`. This output is committed and pushed
with every session. Any change to a JSDoc comment or contract interface produces a diff
across multiple HTML files in CI, making PRs harder to review.

Generated artifacts belong in `.gitignore` or a dedicated `gh-pages`-style deployment
branch, not in the primary working tree.

**Recommended fix:**

1. Add `docs/api/` to `.gitignore`
2. Optionally add a CI job that deploys the generated docs to GitHub Pages from the
   `gh-pages` branch, or keep them ephemeral (generated, inspected locally, discarded)
3. Remove the committed `docs/api/` directory from tracking: `git rm -r --cached docs/api/`

The `npm run docs` CI step then becomes a validation-only step (confirms TypeDoc generates
without errors) rather than producing committed artifacts.

#### L6 — `useTerminal` receives `initialCommands` reference — could retrigger boot on parent re-render

**File:** `src/components/hooks/useTerminal.ts:18`  
**Severity:** Low  
**Category:** Correctness / React Patterns

`useEffect` depends on `[initialCommands]`. `BOOT_COMMANDS` is a module-level `readonly`
array (stable reference), so in practice no re-trigger occurs. However, if a caller ever
passes an inline array literal (`commands={[...]}` in JSX), the effect re-runs on every
render and the terminal reboots infinitely.

The hook has no guard against this. `Terminal.tsx` mitigates it by accepting the array as
a prop (not creating it inline), but the invariant is implicit.

**Recommended fix:** Stabilize `initialCommands` with `useRef` inside the hook so the
effect only runs once regardless of reference identity:

```ts
const initialRef = useRef(initialCommands)
// useEffect depends on [] (empty) — initial commands captured at mount
```

## Architecture Observations

### AO1 — `sitemap.ts` is a static single-entry file with no growth path

`src/app/sitemap.ts` generates one URL. For a portfolio that adds blog posts, case studies,
or sub-pages, the sitemap will need to dynamically enumerate routes. The current pattern
(hardcoded array) doesn't scale, but this is appropriate for the current single-page
structure. **No action required now** — note it as a future consideration.

### AO2 — React Compiler opt-in via Babel plugin is undocumented in the codebase

`next.config.ts` enables the React Compiler via `babel-plugin-react-compiler`. The compiler
auto-memoizes components and hooks — but `ErrorBoundary` is a class component, which the
compiler cannot optimize. This is noted in the 2026-04-27 audit (L2) and remains true.

For `ErrorBoundary`, manual `shouldComponentUpdate` or `React.memo` wrappers on child
components can fill the gap. As a class component wrapping only `TopologyLoader`, the
performance impact is minimal. **No urgent action required.**
