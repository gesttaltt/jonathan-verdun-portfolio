#### H1 — `(es)/layout.tsx` metadata incomplete after H4 split

**Severity:** High  
**File:** `src/app/(es)/layout.tsx:13-24`

The Spanish root layout introduced by H4 is missing `alternates` (hreflang links), a `twitter`
card block, and has a reduced `openGraph` config compared to the English layout. This causes
Spanish routes to emit no `<link rel="alternate">` hreflang tags and no Twitter card metadata,
harming both SEO and social sharing.

**Missing from ES layout vs EN layout:**

```ts
// MISSING:
alternates: {
  canonical: `${siteConfig.url}/es/`,
  languages: { en: siteConfig.url, es: `${siteConfig.url}/es/` },
},
// MISSING:
twitter: {
  card: 'summary_large_image',
  title: '...',
  description: '...',
  creator: siteConfig.socialLinks.twitter.handle,
},
// INCOMPLETE openGraph — missing url, images
```

**Fix:** Add the missing fields to `(es)/layout.tsx`. Also consider extracting a
`buildBaseMetadata(lang)` factory in `src/lib/metadata.ts` to keep both layouts in sync going
forward (see M1).

#### H2 — Boot sequence `help` output lists fewer commands than are actually available

**Severity:** High  
**File:** `src/lib/contracts/TerminalContract.ts:29` and `src/lib/i18n/es.ts:166`

`BOOT_COMMANDS[2].output` (English) says:

```
Available commands: help, about, projects, contact, clear
```

But `INTERACTIVE_COMMANDS` exposes `help, about, projects, research, contact, clear` (including
`ls projects` as an alias). The boot output actively lies about what the terminal can do — `research`
and `ls projects` are silently omitted. The Spanish boot sequence has the same problem (`ayuda`
output on boot says 5 commands; `interactive.ayuda` lists 6).

**Fix:**

```ts
// TerminalContract.ts — boot help output must match interactive help output:
output: 'Available commands: help, about, projects, research, ls projects, contact, clear',
```

And in `es.ts`:

```ts
output: 'Comandos disponibles: ayuda, sobre, proyectos, investigacion, ls proyectos, contacto, limpiar',
```

#### M6 — OG image `readFile` has no error handling — crashes build if font file missing

**Severity:** Medium  
**File:** `src/app/(en)/opengraph-image.tsx:12` · `src/app/(es)/es/opengraph-image.tsx:12`

Both OG image routes call `readFile(join(process.cwd(), 'public/fonts/JetBrainsMono-Bold.ttf'))`
without a try-catch. If the font file is absent (e.g. first clone before `npm install`, or a
`public/` misconfiguration), Next.js will throw an unhandled exception and the route returns 500
instead of a graceful fallback.

**Fix:**

```ts
let fontData: Buffer | undefined
try {
  fontData = await readFile(join(process.cwd(), 'public/fonts/JetBrainsMono-Bold.ttf'))
} catch {
  // fallback: render without custom font
}
// pass fonts: fontData ? [{ name: 'JetBrains Mono', data: fontData, ... }] : []
```

#### L4 — `error.tsx` at `app/` root lacks `<html>/<body>` — parallel to `not-found.tsx`

**Severity:** Low  
**File:** `src/app/error.tsx`

`not-found.tsx` was updated during H4 to include `<html>/<body>` because there is no longer a
root layout above it. `error.tsx` at the same level has the same structural position. If a
layout-level error occurs and Next.js renders `app/error.tsx` outside of a route group context,
it will render without a document wrapper.

For page-level errors (the common case), `error.tsx` is rendered within the route group layout
context and `<html>/<body>` are provided. For layout-level errors you'd need `global-error.tsx`.

**Fix:** Add `src/app/global-error.tsx` (with `<html>/<body>`) as a safety net for layout
failures, and leave `error.tsx` for page-level errors:

```tsx
// app/global-error.tsx
'use client'
export default function GlobalError({ reset }: { reset: () => void }) {
  return (
    <html lang="en">
      <body>
        <div>
          Something went wrong. <button onClick={reset}>Try again</button>
        </div>
      </body>
    </html>
  )
}
```

#### L7 — Spanish `LS_PROYECTOS_OUTPUT` is a private const — not DRY with English

**Severity:** Low  
**File:** `src/lib/i18n/es.ts:3-5`

English uses `LS_PROJECTS_OUTPUT` exported from `TerminalContract.ts`; Spanish declares its own
private `LS_PROYECTOS_OUTPUT`. The two constants list the same projects (different language) but
are decoupled — if a project is added/removed, `es.ts` requires a separate manual update.

**File:** `jest.config.mjs`

Both new group layouts render `<html>/<body>` tags. The layout test exercises them but emits

test results.

#### L10 — `console.error` noise in CI from layout + not-found jsdom rendering

**Severity:** Low  
**File:** `src/__tests__/layout.test.tsx` · `src/__tests__/not-found.test.tsx`

Both files render components that contain `<html>` as the root element. jsdom logs:

```
In HTML, <html> cannot be a child of <div>.
```

This appears in CI test output and can obscure real errors.

**Fix:**

```ts
// Add to layout.test.tsx and not-found.test.tsx beforeAll:
beforeAll(() => {
  jest.spyOn(console, 'error').mockImplementation((msg) => {
    if (typeof msg === 'string' && msg.includes('cannot be a child')) return
    console.warn(msg)
  })
})
afterAll(() => jest.restoreAllMocks())
```
