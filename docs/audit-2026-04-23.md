## 5. Design Review â€” Visual Density & Perceived Breathing Room

**Date:** 2026-04-23  
**Focus:** Minimalism, scroll rhythm, and visual hierarchy

### 5.1 Pain Points Identified

#### Sticky Sidebar Competing Across All Sections

The sidebar's `sticky top-8` caused both cards (Engineering Constraints + Contact Callout) to remain pinned while the user scrolled through three unrelated sections â€” Projects, Architecture, and QA Philosophy. The sidebar had no contextual relationship with what was being read at any given scroll position, creating persistent visual competition and a sense of clutter that worsened the further down the page the user scrolled.

**Fix applied:** Removed `sticky top-8`. The sidebar now scrolls naturally, appearing and reading as part of the page flow rather than a floating overlay.

#### No Scroll-Triggered Hierarchy

All sections â€” Projects, Architecture, QA Philosophy, Sidebar, Footer â€” loaded at full opacity and equal visual weight simultaneously. There was no progressive reveal cue to guide the reader's attention or signal a scroll-based hierarchy. The page read as a flat wall of content rather than a layered narrative.

**Fix applied:** Added `FadeInSection` (`components/FadeInSection.tsx`) â€” a lightweight `whileInView` client wrapper with `viewport={{ once: true }}`. Applied to:

- Architecture section (no delay)
- QA Philosophy section (no delay)
- Sidebar constraints card (0.1s delay)
- Sidebar contact callout (0.2s delay)
- BioinformaticsGraphic footer (no delay)

The Projects section was intentionally excluded â€” `ProjectCard` already carries its own per-card `whileInView` stagger via `fadeUpVariants`.

#### QA Specs Rendered as Static Server Markup

The three QA Philosophy spec items were plain `<div>` elements with no animation, making them visually inert compared to the animated project cards they sit below. The contrast in interactivity between sections undermined consistency.

**Fix applied:** Extracted into `QAPhilosophyGrid` (`components/QAPhilosophyGrid.tsx`) â€” a client component that staggers each spec item into view with a 0.12s delay offset per item using `whileInView`.

### 5.2 Components Added

## 10. CI Pipeline Repair

**Date:** 2026-04-25
**Focus:** GitHub Actions pipeline producing build failures on every push
**Commit:** `fix(ci): resolve build failures blocking GitHub Actions pipeline`
**Result:** Pipeline went from `failure` to `success` â€” all jobs (lint, type-check, test, build) green on Node 20.x and 22.x matrix.

### 10.1 Root Causes Identified

Three independent build failures were blocking CI. None were caught locally until `npm run build` was run explicitly (lint-only CI passes masked them).

#### Failure 1 â€” Unused imports in `ProjectCard.tsx`

`GitBranch` and `ShieldCheck` were imported from `lucide-react` but not rendered anywhere in the component. TypeScript strict mode + ESLint `no-unused-vars` caused a build error.

**Fix:** Removed both from the import statement. Final imports: `{ ExternalLink, Terminal }`.

#### Failure 2 â€” Unused import in `contracts-extended.test.tsx`

`import * as fc from 'fast-check'` was present after the `validateEpitope` property-based test was removed when `BioinformaticsContract.ts` was cleaned up. `fc` was declared but never referenced, triggering the same ESLint rule.

**Fix:** Removed the import entirely. The file now only imports `BioinformaticsService`.

#### Failure 3 â€” RSC boundary violation in `SectionHeader.tsx` (primary build failure)

`SectionHeader` was converted to a Client Component (`'use client'`) in the scroll-rhythm refactor (Section 6) to enable Framer Motion `whileInView` animation. Its `icon` prop remained typed as `icon: LucideIcon` â€” a component function reference.

`page.tsx` is a Server Component (RSC). It passed `icon={Code2}`, `icon={Database}`, and `icon={ShieldCheck}` to `SectionHeader`. React 19 / Next.js 16 enforces that **functions cannot be serialized across the RSC â†’ Client Component boundary**. The build error was:

```
Error: Functions cannot be passed directly to Client Components unless you explicitly expose it by marking it with "use server". Or maybe you meant to call this function rather than return it.
```

**Fix:** Changed the prop signature in `SectionHeader.tsx`:

- Type: `icon: LucideIcon` â†’ `icon: React.ReactNode`
- Render: `<Icon className="h-5 w-5" />` â†’ `{icon}`
- Removed `import type { LucideIcon }` (now unused)

Updated all three call sites in `page.tsx` to pass pre-rendered JSX elements instead of function references:

```tsx
// Before (function reference â€” fails RSC serialization)
<SectionHeader icon={Code2} ... />

// After (pre-rendered JSX â€” safe to serialize)
<SectionHeader icon={<Code2 className="h-5 w-5" />} ... />
```

### 10.3 Architecture Note â€” RSC Boundary Pattern

This failure establishes a clear rule for this codebase: **RSC â†’ Client Component props must be serializable**. Functions, class instances, and component references are not serializable. The canonical solutions are:

1. **Pass `React.ReactNode`** â€” render the JSX in the RSC, pass the result (pre-rendered, serializable). Used here.
2. **Move icon selection into the Client Component** â€” accept a string key (`color: 'purple'`) and map to the icon internally.
3. **Remove `'use client'`** â€” if the animation is the only reason for the directive, consider whether it can be replaced with a CSS alternative.

Option 1 is the least invasive when the parent is already rendering the icon and only the wrapper needs to be a client island.
