## 2. Major Improvements Verified

### Architecture & Performance

- **RSC Shell Implementation:** `page.tsx` is no longer a client component. It is now a Server Component by default, enabling better streaming and reduced client-side hydration. Interactive elements are isolated as client "islands" (`HeroHeader`, `Terminal`, `ProjectGallery`).
- **Lazy Loading WebGL:** `InteractiveTopology` is now lazy-loaded via `next/dynamic({ ssr: false })` using a `TopologyLoader` component. This unblocks the initial First Contentful Paint (FCP).
- **Mobile Optimization:** `PAdicBackground` now dynamically scales particle count (60 → 12) and disables the expensive 2000px rotating SVG on mobile devices.
- **Ultra-wide Support:** Added `2xl:max-w-[1440px]` constraint to the main content container to prevent layout over-stretch.

### Accessibility (P0s Resolved)

- **Motion Preference Awareness:** `MotionProvider` now uses `<MotionConfig reducedMotion="user">`, globally handling `prefers-reduced-motion` for all Framer Motion animations.
- **Canvas ARIA:** `InteractiveTopology` canvas now includes `role="img"` and a descriptive `aria-label`.
- **Mobile Touch Interactivity:** `InteractiveTopology` now uses `touch-action: none` to ensure Three.js can capture pointer events on mobile without browser scrolling interference.

### Content Integrity & Grounding

- **Real Project Data:** Placeholder projects ("Epitope-Scanner", "Chaos-Kube") have been replaced with real, linkable repositories:
  - `qa-arxiv-mobile`
  - `Functionome-Atlas`

- **Grounded Services:** `BioinformaticsService` and `DataEngineeringService` now link to verified GitHub repositories (`hiv-antigen-ai`, `codon-encoder-api`, `work-hours-automated-reports`, etc.). Placeholder `return sequence.includes('ATG')` logic has been removed.
- **Metric Verification:** Fabricated "100% Coverage" stats were removed in favor of real, context-driven metrics (e.g., "120× speed", "1,622 Functionomes").

## 5. Design Review — Visual Density & Perceived Breathing Room

**Date:** 2026-04-23  
**Focus:** Minimalism, scroll rhythm, and visual hierarchy

### 5.1 Pain Points Identified

#### Sticky Sidebar Competing Across All Sections

The sidebar's `sticky top-8` caused both cards (Engineering Constraints + Contact Callout) to remain pinned while the user scrolled through three unrelated sections — Projects, Architecture, and QA Philosophy. The sidebar had no contextual relationship with what was being read at any given scroll position, creating persistent visual competition and a sense of clutter that worsened the further down the page the user scrolled.

**Fix applied:** Removed `sticky top-8`. The sidebar now scrolls naturally, appearing and reading as part of the page flow rather than a floating overlay.

#### No Scroll-Triggered Hierarchy

All sections — Projects, Architecture, QA Philosophy, Sidebar, Footer — loaded at full opacity and equal visual weight simultaneously. There was no progressive reveal cue to guide the reader's attention or signal a scroll-based hierarchy. The page read as a flat wall of content rather than a layered narrative.

**Fix applied:** Added `FadeInSection` (`components/FadeInSection.tsx`) — a lightweight `whileInView` client wrapper with `viewport={{ once: true }}`. Applied to:

- Architecture section (no delay)
- QA Philosophy section (no delay)
- Sidebar constraints card (0.1s delay)
- Sidebar contact callout (0.2s delay)
- BioinformaticsGraphic footer (no delay)

The Projects section was intentionally excluded — `ProjectCard` already carries its own per-card `whileInView` stagger via `fadeUpVariants`.

#### QA Specs Rendered as Static Server Markup

The three QA Philosophy spec items were plain `<div>` elements with no animation, making them visually inert compared to the animated project cards they sit below. The contrast in interactivity between sections undermined consistency.

**Fix applied:** Extracted into `QAPhilosophyGrid` (`components/QAPhilosophyGrid.tsx`) — a client component that staggers each spec item into view with a 0.12s delay offset per item using `whileInView`.

### 5.2 Components Added

| Component              | Type          | Purpose                                                                      |
| ---------------------- | ------------- | ---------------------------------------------------------------------------- |
| `FadeInSection.tsx`    | Client island | Reusable scroll-triggered fade-up wrapper, `once: true`, 450ms ease-out      |
| `QAPhilosophyGrid.tsx` | Client island | Per-item stagger for QA spec cards, replaces inline server map in `page.tsx` |

## 10. CI Pipeline Repair

**Date:** 2026-04-25
**Focus:** GitHub Actions pipeline producing build failures on every push
**Commit:** `fix(ci): resolve build failures blocking GitHub Actions pipeline`
**Result:** Pipeline went from `failure` to `success` — all jobs (lint, type-check, test, build) green on Node 20.x and 22.x matrix.

### 10.1 Root Causes Identified

Three independent build failures were blocking CI. None were caught locally until `npm run build` was run explicitly (lint-only CI passes masked them).

#### Failure 1 — Unused imports in `ProjectCard.tsx`

`GitBranch` and `ShieldCheck` were imported from `lucide-react` but not rendered anywhere in the component. TypeScript strict mode + ESLint `no-unused-vars` caused a build error.

**Fix:** Removed both from the import statement. Final imports: `{ ExternalLink, Terminal }`.

#### Failure 2 — Unused import in `contracts-extended.test.tsx`

`import * as fc from 'fast-check'` was present after the `validateEpitope` property-based test was removed when `BioinformaticsContract.ts` was cleaned up. `fc` was declared but never referenced, triggering the same ESLint rule.

**Fix:** Removed the import entirely. The file now only imports `BioinformaticsService`.

#### Failure 3 — RSC boundary violation in `SectionHeader.tsx` (primary build failure)

`SectionHeader` was converted to a Client Component (`'use client'`) in the scroll-rhythm refactor (Section 6) to enable Framer Motion `whileInView` animation. Its `icon` prop remained typed as `icon: LucideIcon` — a component function reference.

`page.tsx` is a Server Component (RSC). It passed `icon={Code2}`, `icon={Database}`, and `icon={ShieldCheck}` to `SectionHeader`. React 19 / Next.js 16 enforces that **functions cannot be serialized across the RSC → Client Component boundary**. The build error was:

```
Error: Functions cannot be passed directly to Client Components unless you explicitly expose it by marking it with "use server". Or maybe you meant to call this function rather than return it.
```

**Fix:** Changed the prop signature in `SectionHeader.tsx`:

- Type: `icon: LucideIcon` → `icon: React.ReactNode`
- Render: `<Icon className="h-5 w-5" />` → `{icon}`
- Removed `import type { LucideIcon }` (now unused)

Updated all three call sites in `page.tsx` to pass pre-rendered JSX elements instead of function references:

```tsx
// Before (function reference — fails RSC serialization)
<SectionHeader icon={Code2} ... />

// After (pre-rendered JSX — safe to serialize)
<SectionHeader icon={<Code2 className="h-5 w-5" />} ... />
```

| --------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| `components/ProjectCard.tsx`            | Removed unused `GitBranch`, `ShieldCheck` imports                                                         |
| `__tests__/contracts-extended.test.tsx` | Removed unused `import * as fc from 'fast-check'`                                                         |

### 10.3 Architecture Note — RSC Boundary Pattern

This failure establishes a clear rule for this codebase: **RSC → Client Component props must be serializable**. Functions, class instances, and component references are not serializable. The canonical solutions are:

1. **Pass `React.ReactNode`** — render the JSX in the RSC, pass the result (pre-rendered, serializable). Used here.
2. **Move icon selection into the Client Component** — accept a string key (`color: 'purple'`) and map to the icon internally.
3. **Remove `'use client'`** — if the animation is the only reason for the directive, consider whether it can be replaced with a CSS alternative.

Option 1 is the least invasive when the parent is already rendering the icon and only the wrapper needs to be a client island.

### Wave 1 — Branded Error Pages (M1 + M2) · Effort: Low · Impact: High

**Why first:** These are the highest visibility risk items that require no pending decisions. If a recruiter or hiring manager miskeys a URL — or if a runtime error surfaces during review — they hit Next.js's raw, unbranded generic fallback. That breaks the design system entirely and signals an incomplete project regardless of how polished the main page is. Both files are self-contained, ~30–40 lines each, and have no dependencies on any other open item.

#### `app/not-found.tsx`

Stateless RSC. No `'use client'` required. Needs to:

- Render within the existing design system (dark background, monospace font, zinc palette)
- Display a clear 404 signal — suggest using a Terminal-style framing to stay on-brand (e.g., `Error: route not found — exit code 404`)
- Include a `<Link href="/">` back to root with the same hover/glow style used by CTAs elsewhere
- Not import heavy client islands (no Three.js, no Framer Motion) — this page must be fast and resilient by definition

Suggested structure:

```
/ root layout wraps it automatically
→ Centered container, max-w-lg
→ Top: monospace error label in red/zinc
→ Middle: 404 in large font (text-8xl, font-bold, text-white)
→ Sub: short message ("This route does not exist.")
→ CTA: "Return to root" link styled as the contact CTA button
```

#### `app/error.tsx`

Must be a Client Component (`'use client'`) — Next.js requires it to receive the `error` and `reset` props. Needs to:

- Accept `{ error: Error & { digest?: string }, reset: () => void }` props (Next.js App Router contract)
- Render a styled fallback that matches the design system
- Expose the `reset()` function via a button so the user can retry without a hard reload
- Not log `error.message` to any external service (no analytics wired up — skip that concern)
- Display `error.digest` in a small monospace label if present — useful for tracing without exposing a stack trace

Suggested structure:

```
→ Centered container
→ Top: monospace "runtime error" label in amber/zinc
→ Middle: short human message ("Something went wrong.")
→ digest display: `<code className="text-xs text-zinc-500">{error.digest}</code>` (conditional)
→ Two actions: "Try again" (calls reset()) + "Return home" (Link to /)
```

**Acceptance criteria:**

- [ ] Navigating to `/anything-invalid` renders the branded 404, not Next.js default
- [ ] Both pages pass `npm run build` without type errors
- [ ] Neither page imports `'use client'` except `error.tsx` (which requires it)

### Wave 2 — Focus Indicators (A3) · Effort: Medium · Impact: High (on-brand)

**Why second:** This is a WCAG 2.1 Level AA failure (2.4.7) — keyboard users cannot track focus on project cards, section headers, or the contact CTA. For a portfolio that positions as QA-focused, having accessibility failures in the production site is a direct credibility issue. A technical interviewer who tabs through the page will notice immediately.

**Scope:** All interactive elements that currently lack `:focus-visible` styles:

- `ProjectCard` — the external link icon button (`<Link>` wrapping `<ExternalLink />`)
- `HeroHeader` — GitHub, LinkedIn, and Ai-Whisperers links; the resume/contact CTA buttons
- `SystemSpecCard` — the repo link at the bottom
- `SectionHeader` — not interactive itself, no change needed
- Contact CTA in sidebar — `<Link href="mailto:...">` in `page.tsx`

**Implementation pattern:** Tailwind utility classes, applied directly at the element level:

```
focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:ring-offset-2 focus-visible:ring-offset-black
```

For icon-only links (GitHub, LinkedIn), the ring must be paired with `rounded-full` on the focus ring to match the button's border-radius. For card-level links, `rounded-lg` offset works.

The `focus-visible` pseudo-class (not `focus`) is intentional — it suppresses the ring on mouse click while preserving it for keyboard navigation, which is the modern WCAG-compliant approach.

**Acceptance criteria:**

- [ ] Tab-navigating through the page shows a visible blue ring on every focusable element
- [ ] Mouse click does not trigger the ring (`:focus-visible` only, not `:focus`)
- [ ] Ring colour (`ring-blue-500`) is consistent with the site's primary accent
