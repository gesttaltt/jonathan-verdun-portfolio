#### C2 — `DefaultCommandProcessor` instantiated on every render in `PortfolioPage`

**File:** `src/components/PortfolioPage.tsx:24`  
**Severity:** Critical  
**Category:** React / Performance

```tsx
const processor = new DefaultCommandProcessor(t.terminal.interactive)
```

This line runs on every render of `PortfolioPage`. `Terminal` receives `processor` as a
prop — every render hands it a new object reference. If `Terminal` (or any of its children)
depends on the prop reference being stable (e.g., in a `useEffect` or `useMemo`
dependency array), this causes unnecessary re-runs.

`useTerminal.ts` does not currently depend on `processor` in an effect, but the pattern
is fragile and will silently break if `useTerminal` is ever extended to re-initialize on
processor change.

**Recommended fix:**

```tsx
const processor = useMemo(
  () => new DefaultCommandProcessor(t.terminal.interactive),
  [t.terminal.interactive]
)
```

Add `useMemo` to the `PortfolioPage.tsx` import list.

#### H1 — "command not found" error string hardcodes English command name

**File:** `src/lib/services/CommandProcessor.ts:20`  
**Severity:** High  
**Category:** i18n Correctness

```ts
return `bash: ${cmd}: command not found. Type 'help' for available commands.`
```

Spanish visitors type `ayuda` (not `help`) to get the command list. When they mistype
any command they see: _"Type 'help' for available commands."_ — pointing to a command that
doesn't exist in their locale. English users type `help`, so the message is wrong only for
Spanish visitors, but that is exactly the route that has i18n.

The fix requires either passing the "not found" template as a translatable string into
`DefaultCommandProcessor`, or resolving the help alias from the commands map:

**Recommended fix:**

```ts
export class DefaultCommandProcessor implements ICommandProcessor {
  private readonly commands: Record<string, string>
  private readonly notFoundMsg: (cmd: string, helpCmd: string) => string

  constructor(
    customCommands?: Record<string, string>,
    notFoundMsg?: (cmd: string, helpCmd: string) => string
  ) {
    this.commands = customCommands ?? INTERACTIVE_COMMANDS
    this.notFoundMsg =
      notFoundMsg ??
      ((cmd, helpCmd) => `bash: ${cmd}: command not found. Type '${helpCmd}' for available commands.`)
  }
```

Then pass `notFoundMsg` from `en.ts` / `es.ts` alongside `interactive`. The help alias
(`'help'` vs `'ayuda'`) is inferred as the key whose value starts with `'Available commands'`
/ `'Comandos disponibles'`, or simply added as a dedicated field in the `Translations.terminal`
type.

#### H4 — `<html lang="en">` hardcoded for all routes; Spanish page has wrong `lang` attribute

**File:** `src/app/layout.tsx:64`  
**Severity:** High  
**Category:** Accessibility / SEO

The shared root layout emits `<html lang="en">` for every route including `/es/`.
Screen readers and browser translation features use the `lang` attribute to select the
correct language engine. A Spanish page with `lang="en"` will be mispronounced by
VoiceOver/NVDA and incorrectly offered for translation by Chrome.

In Next.js App Router with `output: 'export'`, the root layout is the only place `<html>`
can be rendered. The standard workaround is **route groups**:

```
src/app/
  (en)/
    layout.tsx          ← <html lang="en">
    page.tsx
  (es)/
    layout.tsx          ← <html lang="es">
    es/
      page.tsx
```

Route groups (parenthesised segment names) are transparent to URL routing but allow each
sub-tree to have its own root layout. This is a meaningful refactor but the correct solution.

**Recommended fix:** Introduce `(en)` and `(es)` route groups, each with their own
root layout providing the correct `lang` attribute.

#### M2 — `LanguageSelector` uses `aria-current="true"` instead of `aria-current="page"`

**File:** `src/components/LanguageSelector.tsx:20,30`  
**Severity:** Medium  
**Category:** Accessibility

```tsx
aria-current={!isEs ? 'true' : undefined}
```

`aria-current` accepts a set of semantic values. For navigation links, the correct value
is `'page'` (signals "this link represents the current page"). `'true'` is the generic
boolean form and is less meaningful to AT users — screen readers announce
"current" rather than "current page".

Per WAI-ARIA 1.2: _"Use `aria-current="page"` to identify the link to the current page
within a set of pagination links."_

**Recommended fix:**

```tsx
aria-current={!isEs ? 'page' : undefined}
```

Also: the wrapper `<div aria-label="Language selector">` is not a landmark. Using
`<nav aria-label="Language selector">` gives it a navigable ARIA landmark role and is
semantically more accurate (it contains navigation links between site languages).

#### M5 — `TerminalContract.ts` is effectively orphaned — no longer the source of truth

**File:** `src/lib/contracts/TerminalContract.ts`  
**Severity:** Medium  
**Category:** Architecture / Dead Code

`TerminalContract.ts` exports `INTERACTIVE_COMMANDS` and `BOOT_COMMANDS`. Before i18n,
these were the sole command sources. Now:

- `PortfolioPage.tsx` always passes `t.terminal.interactive` to `DefaultCommandProcessor`,
  overriding `INTERACTIVE_COMMANDS`.
- `PortfolioPage.tsx` passes `t.terminal.boot` to `Terminal`, superseding `BOOT_COMMANDS`.
- `TerminalContract.ts` is only imported by `CommandProcessor.ts` as a fallback (used when
  `Terminal` is rendered without a `processor` prop — which never happens in the live app)
  and by `CommandProcessor.test.ts` for test assertions.

The contract is now test infrastructure rather than production data. If English translations
in `en.ts` diverge from `TerminalContract.ts`, tests pass but the live page shows
different content.

**Recommended fix (two options):**

_Option A:_ Have `i18n/en.ts` import and re-export `INTERACTIVE_COMMANDS` / `BOOT_COMMANDS`
from `TerminalContract.ts` directly, keeping one source of truth for the English copy.

_Option B:_ Delete `TerminalContract.ts`. Update `CommandProcessor.ts` to throw if no
`customCommands` are passed (removing the fallback). Update tests to use `en.ts` data.

#### M6 — `sitemap.ts` fallback date will become stale

**File:** `src/app/sitemap.ts:7`  
**Severity:** Medium  
**Category:** SEO / Maintenance

```ts
const lastModified = new Date(process.env.SITE_LAST_MODIFIED ?? '2026-04-30')
```

The hardcoded fallback `'2026-04-30'` is today's date and will never advance unless
manually updated. Search engines use `lastModified` to prioritise re-crawl scheduling.
A permanently stale date tells crawlers the site has not changed, reducing crawl frequency.

**Recommended fix:** Replace the hardcoded fallback with `new Date()`:

```ts
const lastModified = process.env.SITE_LAST_MODIFIED
  ? new Date(process.env.SITE_LAST_MODIFIED)
  : new Date()
```

This ensures every build without an explicit env variable records the actual build date.

#### L6 — `robots.txt` is a static file; no programmatic generation from `siteConfig`

**File:** `public/robots.txt`  
**Severity:** Low  
**Category:** Maintainability

`public/robots.txt` hardcodes `Sitemap: https://jonathanverdun.com/sitemap.xml`. If the
domain in `siteConfig.url` ever changes, `robots.txt` must be manually updated. The
`sitemap.ts` and `layout.tsx` both use `siteConfig.url` dynamically.

Next.js supports `app/robots.ts` which generates `robots.txt` at build time. This would
let `robots.ts` reference `siteConfig.url` and stay in sync automatically.

**Recommended fix:** Create `src/app/robots.ts` and remove `public/robots.txt`:

```ts
import type { MetadataRoute } from 'next'
import { siteConfig } from '@/lib/siteConfig'

export default function robots(): MetadataRoute.Robots {
  return {
    rules: { userAgent: '*', allow: '/' },
    sitemap: `${siteConfig.url}/sitemap.xml`,
  }
}
```

#### L7 — `SiteFooter` year value recalculated on every render

**File:** `src/components/SiteFooter.tsx:30`  
**Severity:** Low  
**Category:** React Patterns

```tsx
const year = new Date().getFullYear()
```

`new Date()` is called on every render. For a static footer this has negligible cost, but
the value is a constant for any given browsing session. In a static export build the value
is baked in at build time and is always correct, but the pattern signals missing
stabilization.

**Recommended fix:** Move `year` outside the component (module-level constant) since it
never changes between renders:

```ts
const YEAR = new Date().getFullYear()
```

Or inside as `useMemo`:

```tsx
const year = useMemo(() => new Date().getFullYear(), [])
```
