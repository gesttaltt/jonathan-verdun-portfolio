## Executive Summary

The portfolio is in a strong baseline state: security headers are correctly configured, LazyMotion bundle splitting is applied, semantic HTML is mostly present, and the test suite is healthy. This audit focuses exclusively on **improvement areas** — issues found in the current codebase that are worth fixing, ordered by impact.

**Three critical findings** require action before the portfolio can be considered complete:

1. Three unused production dependencies inflate `node_modules` and signal that the dependency list has drifted from reality.
2. The QA philosophy section claims property-based fuzzing with `fast-check` — but no such tests exist. This is a false claim on a QA-focused portfolio where credibility is the product.
3. Coverage thresholds are set at 30%, far below actual coverage of 76%. They provide zero protection against regressions.

### C2 — QA Philosophy Claims fast-check Fuzzing — No Property-Based Tests Exist

**Files:** `src/lib/contracts/QAContract.ts`, `src/__tests__/` (absence)

`QAContract.ts` presents two claims that are visible on the live portfolio:

- **Option B:** Update `QAContract.ts` to remove the fast-check claim and replace `layer: 'property-based'` with something that accurately reflects the current test suite (e.g., `layer: 'integration'` or add a new specification for it with `status: 'planned'`).

### C3 — Coverage Thresholds Far Below Actual Coverage — No Regression Protection

**File:** `jest.config.mjs:22-29`

```js
coverageThreshold: {
  global: {
    lines: 30,      // actual: 77.2%
    functions: 25,  // actual: 76.56%
    branches: 20,   // actual: 70.17%
    statements: 30, // actual: 76%
  },
},
```

A 40-point gap between threshold and actual means coverage could fall from 76% to 35% before CI fails. The thresholds are nominal placeholders — they guard against total collapse, not regression.

**Fix:** Raise thresholds to actual-minus-5 (to allow for minor fluctuation and the WebGL ceiling):

```js
coverageThreshold: {
  global: {
    lines: 72,
    functions: 70,
    branches: 65,
    statements: 72,
  },
},
```

Branch threshold is set lower than lines/statements because the WebGL components (`InteractiveTopology`, `TopologyMesh`) cannot be covered without `jest-canvas-mock`, so their branch coverage is structurally zero.

### H2 — `autoFocus` on Terminal Input Causes Page Scroll and Focus Order Violation

**File:** `src/components/Terminal.tsx:88`

```tsx
<input
  ref={inputRef}
  ...
  autoFocus   // ← problem
  ...
/>
```

`autoFocus` moves browser focus to the terminal input immediately on page load. This:

1. **Scrolls the page past the hero header** — visitors on mobile see the terminal, not the name/tagline.
2. **Violates WCAG 2.4.3 (Focus Order)** — the focus sequence should follow the reading/visual order. The `HeroHeader` appears above the terminal in the DOM but is never focused first.
3. **Is redundant** — `Terminal.tsx:39` has `onClick={() => !isBooting && inputRef.current?.focus()}`, so clicking anywhere in the terminal already focuses the input. `autoFocus` only fires once during boot when the input isn't interactive anyway (`isBooting` is `true` and the input isn't rendered yet).

Wait — looking more carefully at the render logic: the input only renders when `!isBooting` (line 76). So `autoFocus` fires at the moment the boot sequence finishes. At that point, the user may have scrolled elsewhere. Programmatic focus will then pull them back to the terminal unexpectedly.

**Fix:** Remove `autoFocus` from the input. The `onClick` handler on the terminal div already handles focus-on-click. If an interactive focus-on-reveal is desired, use `inputRef.current?.focus()` inside the `setIsBooting(false)` callback in `useTerminal`, but only if `document.activeElement === document.body` (i.e., nothing else has focus yet).

### H3 — `BioinformaticsGraphic` Hover Interaction Has No Keyboard Equivalent

**File:** `src/components/BioinformaticsGraphic.tsx:79-90`

```tsx
<m.div
  className="... cursor-default ..."
  whileHover={{ scale: 1.1, color: 'var(--accent)' }}
  ...
>
  <span>{codon}</span>
  <m.div
    className="absolute inset-0 bg-blue-500/20"
    whileHover={{ x: '100%' }}   // scanning effect
  />
</m.div>
```

The DNA codon chips (`ATG`, `GCT`, etc.) animate on hover but have `cursor-default`, no `tabIndex`, and no `role`. WCAG 1.4.13 (Content on Hover) applies when hover reveals or changes content — the scanning effect and color change count. Keyboard users have no way to trigger this behavior.

**Fix options:**

- **Option A (simplest):** Remove `whileHover` and make the codons purely static/decorative. The SVG DNA helix animation below already provides visual interest.
- **Option B:** Add `tabIndex={0}`, `role="img"`, and `aria-label={codon}` to each codon chip, then use `whileFocus` alongside `whileHover` so keyboard users see the same effect.

Given the codons are decorative (standard genetic code triplets, not interactive data), Option A is recommended.

### H5 — `key={index}` in Four Static Arrays

**Files:** `src/app/page.tsx:64`, `src/components/QAPhilosophyGrid.tsx:13`, `src/components/BioinformaticsGraphic.tsx:57,75`

```tsx
// page.tsx
{QA_PHILOSOPHY.constraints.map((c, i) => (
  <div key={i} ...>

// QAPhilosophyGrid.tsx
{QA_PHILOSOPHY.specifications.map((spec, i) => (
  <m.div key={i} ...>

// BioinformaticsGraphic.tsx
```

### M3 — Terminal `onClick` on `div` Lacks Keyboard Trigger

**File:** `src/components/Terminal.tsx:39`

```tsx
<div
  ...
  onClick={() => !isBooting && inputRef.current?.focus()}
>
```

This outer `div` handles clicks to focus the input, but `div` elements are not in the tab order and do not receive keyboard events. A keyboard user who tabs to the terminal area (the input inside it will receive focus directly) is fine, but a user who clicks the chrome area (title bar, scrollable log region) and expects keyboard input will find the click works but tab/enter don't trigger the `onClick`.

**Fix:** The `onClick` is on the outer container, primarily to make the "dead space" around the input clickable. The input itself is inside the terminal and can receive focus normally via keyboard navigation. The `onClick` can stay as-is since it's a convenience UX enhancement. However, to be fully correct, either add `role="group"` or `tabIndex={-1}` so the div is programmatically focusable when needed.

### M4 — `useTerminal` Boot Function Is Unnecessarily `async`

**File:** `src/components/hooks/useTerminal.ts:22`

```ts
const boot = async () => {
  for (const cmd of initialCommands) {
    // only setTimeout — no await
    const timeout = setTimeout(...)
    timeouts.push(timeout)
  }
  const finishTimeout = setTimeout(...)
  timeouts.push(finishTimeout)
}

boot()   // returned Promise is discarded
```

`boot` is declared `async` but never uses `await`. The returned `Promise` is discarded (no `await boot()` or `.catch()`). This has no runtime effect because the function is synchronous in practice, but it:

- Misleads readers into thinking there's async I/O
- Suppresses any future unhandled rejection if someone adds `await` inside
- Could confuse linters (`@typescript-eslint/no-floating-promises`)

**Fix:** Remove `async` from `boot`. The function is synchronous — `setTimeout` is callback-based, not Promise-based.

### M5 — `sitemap.ts` Uses `new Date()` — Non-Deterministic Build Output

**File:** `src/app/sitemap.ts:8`

```ts
lastModified: new Date(),
```

Every build generates a new `lastModified` timestamp, causing `sitemap.xml` to report the page as modified even when no content changed. Search engine crawlers may waste crawl budget on unchanged content.

**Fix:** Use a fixed date tied to the last meaningful content update, or derive it from an environment variable set in the build pipeline:

```ts
lastModified: new Date(process.env.SITE_LAST_MODIFIED ?? '2026-04-27'),
```

Alternatively, use `git log` to derive the date automatically via a build script.

### M6 — `ErrorBoundary` Has No Reset Mechanism

**File:** `src/components/ErrorBoundary.tsx`

The `ErrorBoundary` catches errors and renders either `props.fallback` or a generic "Component Error" card. Once an error is caught, the user's only recourse is a full page refresh. The `fallback` prop allows custom rendering, but there's no `onReset` prop or internal reset button.

Compare with `error.tsx` which correctly provides a `reset()` callback from Next.js.

For the current use case (wrapping `TopologyLoader` — WebGL), a full refresh is appropriate. However, if `ErrorBoundary` is ever reused for other components, the lack of reset makes it incomplete.

**Fix (optional):** Add a `onReset?: () => void` prop. In the default fallback render, add a reset button that calls `this.setState({ hasError: false, error: null })` and then `props.onReset?.()`. This is a small addition that makes the component more robust.

### L2 — `reactCompiler: true` Does Not Apply to Class Components

**File:** `next.config.ts:20`, `src/components/ErrorBoundary.tsx`

The React Compiler (`reactCompiler: true`) automatically memoizes function components. `ErrorBoundary` is a class component and is explicitly excluded from compiler optimization. This is not a bug — class components are valid React — but it means `ErrorBoundary` will never benefit from automatic memoization, and the React team has signaled that class components are in maintenance mode.

**Note only:** No action required unless `ErrorBoundary` is refactored to use the `use-error-boundary` hook pattern or a functional equivalent.

### L5 — CSP Contains `'unsafe-eval'` Due to Three.js Shader Compilation

**File:** `next.config.ts:8`

```js
"script-src 'self' 'unsafe-inline' 'unsafe-eval'",
```

`'unsafe-eval'` is required for Three.js GLSL shader compilation in Firefox (Chromium compiles shaders without `eval`). This weakens the CSP meaningfully — `eval` allows arbitrary code execution from strings.

**Known trade-off:** No short-term fix without replacing Three.js or pre-compiling shaders to WGSL (which requires a different rendering architecture). Document this in a comment explaining which browser/library requires it:

```js
// 'unsafe-eval': required by Three.js GLSL shader compilation in Firefox.
// To remove: pre-compile shaders or migrate to WebGPU/WGSL.
"script-src 'self' 'unsafe-inline' 'unsafe-eval'",
```

### A1 — Bioinformatics Section Is Disconnected from the Bioinformatics Contract

`BioinformaticsContract.ts` exports `BioinformaticsService.getResearchSpecs()` with two research specs (HIV antigen AI, Codon Encoder API). This data is never rendered anywhere in the UI — the `BioinformaticsGraphic` component is decorative (hardcoded codons), and there is no section that lists the bioinformatics research specs the way the Architecture section lists `DataEngineeringService.getSystemSpecs()`.

**File:** `src/app/layout.tsx:44`
